CLI Help

When using compose.mk directly, help not only works, it goes beyond simply listing available targets, and actually parses and displays documentation per target, and per target-namespace. (Under the hood, a dockerized version of charmbracelete/glow renders markdown.).

As seen in the animation, there are several ways to invoke the help for compose.mk. See the examples below:

# list all targets 
$ ./compose.mk help

# help for a single target 
$ ./compose.mk help docker.run

# help for a module namespace with many targets
$ ./compose.mk help docker

Help For Project Makefiles

To enable help for your project Makefile, it is helpful to use include compose.mk somewhere near the top. By itself, this enables target listing with make help, including both the inherited targets from compose.mk as well as the new targets that are project specific.

However for technical reasons1 related to default argument parsing by make, invocations like make help docker cannot work with project makefiles. A work-around is to directly provide arguments like so:

$ make help search=docker.run

Note that for help to render properly, your targets need to use the following python-docstring inspired style for comments, and then help can be pulled automatically from source.


include compose.mk 

my-target:
    @# Comment line one
    @# **Markdown is supported.**
    echo your implementation here

my-parametric-target/%:
    @# Comment line one
    @# **Markdown is supported.**
    echo your implementation here

Implementation Details

The implementation for help uses mk.help.* family of targets, and if you want to customize help or error messages they might be useful.

See the API for:

  • mk.help
  • mk.help.block/<arg>
  • mk.help.module/<arg>
  • mk.help.search/<arg>
  • mk.help.target/<arg>

Heavy lifting for parsing Makefile metadata comes from a dockerized version of the pynchon tool, which can parse help as well as target-prerequisites / DAGs.
